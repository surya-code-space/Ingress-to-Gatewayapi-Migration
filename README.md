# Gateway API Demo — Complete Guide

Zero-downtime migration from **nginx Ingress → Kubernetes Gateway API**, with live dual-path monitoring and post-migration feature demos: canary traffic splitting, CORS headers, header-based routing, URL rewrite with timeouts, and cross-namespace routing.

---

## What This Demo Does

This demo simulates a real enterprise migration at the scale Ford Platform Engineering runs — except compressed into a local kind cluster you can run on your laptop.

**The story:**

1. You have a running app served by legacy nginx Ingress (`gpen-ingress`) — this is the "before" state
2. ingress-nginx was officially retired March 24, 2026 — no more patches, no more CVE fixes
3. You use the official `ingress2gateway` CLI to auto-convert your Ingress to HTTPRoute YAML
4. You run **both** nginx and Envoy Gateway simultaneously and compare responses in real time
5. Once confident, you delete the Ingress — 100% traffic switches to Gateway API with zero downtime
6. You then layer on features that were impossible with annotations: canary weights, CORS, header routing, URL rewrite, cross-namespace backends — all in the HTTPRoute spec

## Script Reference

### 01-setup.sh — Install Everything & Create Cluster

**Run once before the demo. Takes 5–8 minutes on first run.**

```bash
cd path/to/gateway-api-talk/demo
chmod +x scripts/*.sh
./scripts/01-setup.sh
```

#### What it does, step by step

**Step 1 — Check Docker is running**

```bash
docker info
```

kind creates Kubernetes nodes as Docker containers. If Docker Desktop is not open, the script exits immediately.

**Step 2 — Install kind (Kubernetes in Docker)**

```bash
brew install kind
kind version     # kind v0.23.0
```

kind creates a full single-node Kubernetes cluster inside a Docker container on your Mac. No cloud account needed.

**Step 3 — Install Helm**

```bash
brew install helm
helm version     # v3.15.0
```

Helm installs Envoy Gateway from an OCI registry in one command.

**Step 4 — Install ingress2gateway v0.3.0 (auto-detects arm64 / amd64)**

```bash
# Apple Silicon binary:
curl -sSL https://github.com/kubernetes-sigs/ingress2gateway/releases/download/v0.3.0/ingress2gateway_Darwin_arm64.tar.gz
ingress2gateway version     # v0.3.0
```

`ingress2gateway` is the official CNCF SIG-Network CLI that reads your live Ingress resources from the cluster and outputs equivalent Gateway API HTTPRoute YAML. You never rewrite migration YAML by hand.

**Step 5 — Create kind cluster named `gpen-demo`**

```bash
kind create cluster --name gpen-demo --config - <<EOF
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  kubeadmConfigPatches:
  - |
    kind: InitConfiguration
    nodeRegistration:
      kubeletExtraArgs:
        node-labels: "ingress-ready=true"
  extraPortMappings:
  - containerPort: 80
    hostPort: 8080
    protocol: TCP
  - containerPort: 443
    hostPort: 8443
    protocol: TCP
EOF
```

Key config:
- `node-labels: ingress-ready=true` — nginx-ingress uses this to schedule its NodePort on this node
- `containerPort 80 → hostPort 8080` — maps port 80 inside Docker to port 8080 on your Mac — how `curl localhost:8080` reaches nginx

**Step 6 — Install nginx-ingress controller (the "before" state)**

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml
kubectl wait --namespace ingress-nginx \
  --for=condition=ready pod \
  --selector=app.kubernetes.io/component=controller \
  --timeout=120s
```

This is ingress-nginx — officially retired March 24, 2026. Its presence is intentional: it is what we migrate away from.

**Step 7 — Deploy sample app**

Three services using `hashicorp/http-echo:latest` (returns a fixed string on port 5678):

| Deployment | Response | Replicas |
|------------|----------|---------|
| `gpen-web` | `🌐 Web Frontend` | 2 |
| `gpen-api-v1` | `🔵 API v1 — stable` | 2 |
| `gpen-api-v2` | `🟢 API v2 — canary` | 1 |

**Step 8 — Apply legacy Ingress resource**

```bash
kubectl apply -f ingress/ingress.yaml
```

Creates `gpen-ingress` (nginx class) routing `gpen.local/` → gpen-web and `gpen.local/api` → gpen-api-v1.

**Step 9 — Install Envoy Gateway v1.3.0**

```bash
helm install eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.3.0 \
  --namespace envoy-gateway-system \
  --create-namespace \
  --wait --timeout 3m
```

Installs the Envoy Gateway controller and all Gateway API CRDs (GatewayClass, Gateway, HTTPRoute, ReferenceGrant).

**Step 10 — Verify /etc/hosts**

```bash
grep gpen.local /etc/hosts
# 127.0.0.1  gpen.local
```

#### After setup — verify the "before" state works

```bash
kubectl port-forward -n ingress-nginx svc/ingress-nginx-controller 8080:80 &

curl -H "Host: gpen.local" http://localhost:8080/
# 🌐 Web Frontend

curl -H "Host: gpen.local" http://localhost:8080/api
# 🔵 API v1 — stable
```

### 02-migrate.sh — Enterprise Migration (6 Phases)

**Terminal 1. Press ENTER between phases.**

```bash
./scripts/02-migrate.sh
```

#### Phase 0 — Pre-flight: Audit & Backup

```bash
# Cluster overview:
kubectl get nodes -o wide
kubectl get pods,svc,ingress -o wide

# Backup all Ingress resources (rollback anchor):
kubectl get ingress -A -o yaml > ingress-backup.yaml

# Preview what ingress2gateway will generate (nothing applied yet):
ingress2gateway print --namespace=default --providers=ingress-nginx
```

The backup file is your emergency restore — if anything goes wrong after cutover:
```bash
kubectl apply -f ingress-backup.yaml   # 5-second rollback
```

#### Phase 1 — Infrastructure: GatewayClass + Gateway (platform team, once)

```bash
kubectl apply -f gateway/01-gateway.yaml
```

This creates two resources:

**GatewayClass** — registers which controller handles Gateways of class `gpen-gateway`:
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: gpen-gateway
spec:
  controllerName: gateway.envoyproxy.io/gatewayclass-controller
```

**Gateway** — the load balancer definition owned by the platform team:
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: gpen-gateway
spec:
  gatewayClassName: gpen-gateway
  listeners:
  - name: http
    protocol: HTTP
    port: 80
```

When applied, Envoy Gateway automatically spins up an Envoy proxy pod and creates a LoadBalancer Service. The script polls until `Programmed: True`:

```bash
kubectl get gateway gpen-gateway -o jsonpath='{.status.conditions[?(@.type=="Programmed")].status}'
# True

kubectl get gateway gpen-gateway -o wide
# NAME           CLASS          ADDRESS   PROGRAMMED   AGE
# gpen-gateway   gpen-gateway             True         2m
```

#### Phase 2 — Convert: ingress2gateway Generates HTTPRoutes

nginx is still live. Migration runs in parallel — nothing breaks.

```bash
# Port-forward nginx (keep it alive):
kubectl port-forward -n ingress-nginx svc/ingress-nginx-controller 8080:80 &

# Verify nginx is healthy before touching anything:
curl -s -o /dev/null -w "%{http_code}" -H "Host: gpen.local" http://localhost:8080/
# 200

# Run ingress2gateway — auto-convert live Ingress to HTTPRoute YAML:
ingress2gateway print --namespace=default --providers=ingress-nginx
```

The script saves output to `gateway/ingress2gateway-converted.yaml`, then patches the `parentRef` from `nginx` → `gpen-gateway`, then applies it:

```bash
kubectl apply -f gateway/ingress2gateway-converted.yaml
kubectl get httproute -o wide
```

**At this point: nginx Ingress is still running AND HTTPRoute is created. No traffic has switched.**

#### Phase 3 — Parallel Run: Both Paths Live Simultaneously

This is the zero-downtime moment. Both controllers route to the same backend pods.

```bash
# Port-forward Envoy Gateway (the new path):
kubectl port-forward -n <eg-namespace> svc/<envoy-svc> 9080:80 &

# Test both paths — they must return identical responses:
curl -s -H "Host: gpen.local" http://localhost:8080/     # nginx: 🌐 Web Frontend
curl -s -H "Host: gpen.local" http://localhost:9080/     # Envoy: 🌐 Web Frontend

curl -s -H "Host: gpen.local" http://localhost:8080/api  # nginx: 🔵 API v1 — stable
curl -s -H "Host: gpen.local" http://localhost:9080/api  # Envoy: 🔵 API v1 — stable
```

**Switch to Terminal 2 now and start the monitor:**

```bash
./scripts/02-migrate-monitor.sh
# Faster: ./scripts/02-migrate-monitor.sh --interval 0.5
```

The script waits 10 seconds then prompts you to press ENTER. Keep the monitor running in Terminal 2.

#### Phase 4 — Validate: Automated Health Comparison

The script runs 20 comparison requests and verifies nginx and Envoy return identical 200 responses:

```
[1/20]  ✔  nginx=200  gateway=200  — match
[2/20]  ✔  nginx=200  gateway=200  — match
...
[20/20] ✔  nginx=200  gateway=200  — match
Result: 20 passed  0 failed  out of 20
✔ All 20/20 requests match — Gateway API path is healthy
✔ Safe to proceed to cutover
```

If any check fails, cutover is blocked and rollback options are shown automatically.

#### Phase 5 — Cutover: Delete Ingress, Gateway API Takes Full Control

```
POINT OF NO RETURN (but rollback is still fast):
Rollback: kubectl apply -f ingress-backup.yaml
Rollback time: ~5 seconds

Proceed with Ingress deletion? [y/N]
```

Type `y`:

```bash
kubectl delete ingress gpen-ingress
```

**100% of traffic now flows through Envoy Gateway.** nginx-ingress controller is still running but has no Ingress to process.

The monitor in Terminal 2 automatically detects this — you will see it transition from `PARALLEL` to `CUTOVER` with a green banner: *nginx=404 (EXPECTED) Envoy=200 (EXPECTED) — This is NOT a mismatch*.

Post-cutover validation runs automatically:
```bash
curl -H "Host: gpen.local" http://localhost:9080/
# 200 → 🌐 Web Frontend

curl -H "Host: gpen.local" http://localhost:9080/api
# 200 → 🔵 API v1 — stable
```

#### Phase 6 — Canary: Native Traffic Split (bonus)

A feature that was impossible with nginx Ingress annotations:

```bash
kubectl apply -f gateway/02-canary.yaml
```

HTTPRoute updated to split `/api` traffic:
- `weight: 90` → `gpen-api-v1` (stable)
- `weight: 10` → `gpen-api-v2` (canary)

Script hits `/api` 20 times and shows distribution:
```
[1]  🔵 API v1 — stable
[2]  🔵 API v1 — stable
[3]  🟢 API v2 — canary   ← CANARY
...
Distribution: v1=18/20  v2=2/20  (target ~18/2)
ℹ weight: 90/10 — impossible with nginx Ingress annotations
```

**Terminal 1. Run after migrate.sh completes.**

#### Phase 1 — CORS: Native Response Headers

**nginx way:** controller-specific annotations (`nginx.ingress.kubernetes.io/enable-cors`, `nginx.ingress.kubernetes.io/cors-origin`) — inconsistent across versions and controllers.

**Gateway API way:** `ResponseHeaderModifier` filter inline in the HTTPRoute spec:

```bash
kubectl apply -f gateway/03-cors.yaml
```

```yaml
filters:
- type: ResponseHeaderModifier
  responseHeaderModifier:
    add:
    - name: "Access-Control-Allow-Origin"
      value: "https://portal.internal"
    - name: "Access-Control-Allow-Methods"
      value: "GET, POST, PUT, DELETE, OPTIONS"
    - name: "Access-Control-Allow-Headers"
      value: "Content-Type, Authorization, X-Request-ID"
    - name: "Access-Control-Max-Age"
      value: "86400"
```

Test:
```bash
curl -sI -H "Host: gpen.local" http://localhost:9080/ | grep -i "access-control"
# Access-Control-Allow-Origin: https://portal.internal
# Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
# Access-Control-Allow-Headers: Content-Type, Authorization, X-Request-ID
# Access-Control-Max-Age: 86400
```

#### Phase 2 — Header Routing: A/B Testing

**nginx way:** Two separate Ingress objects + canary annotations — fragile, not portable between controllers.

**Gateway API way:** A `headers` match rule inside the HTTPRoute — evaluated before the default rule:

```bash
kubectl apply -f gateway/04-header-routing.yaml
```

```bash
# No header — always v1:
curl -s -H "Host: gpen.local" http://localhost:9080/api
# 🔵 API v1 — stable

# With X-Version: v2 — always v2:
curl -s -H "Host: gpen.local" -H "X-Version: v2" http://localhost:9080/api
# 🟢 API v2 — canary
```

Use case: internal beta users get a special header injected by auth middleware, routing them to the new version with zero DNS changes and zero re-deployments.

#### Phase 3 — URL Rewrite + Per-Route Timeouts

**nginx way:** `nginx.ingress.kubernetes.io/rewrite-target` annotation with regex capture groups. No per-route timeout support at all.

**Gateway API way:** `URLRewrite` filter + `timeouts` block in the same rule:

```bash
kubectl apply -f gateway/05-rewrite-timeout.yaml
```

```yaml
- matches:
  - path: { type: PathPrefix, value: /v1/products }
  filters:
  - type: URLRewrite
    urlRewrite:
      path:
        type: ReplacePrefixMatch
        replacePrefixMatch: /api
  timeouts:
    request: 3s          # total request budget
    backendRequest: 2s   # backend must respond within 2s
  backendRefs:
  - name: gpen-api-v1
    port: 80
```

Test:
```bash
# Client sends /v1/products — backend receives /api:
curl -s -H "Host: gpen.local" http://localhost:9080/v1/products
# 🔵 API v1 — stable

# Original /api still works:
curl -s -H "Host: gpen.local" http://localhost:9080/api
# 🔵 API v1 — stable
```

The backend never sees `/v1/products`. Envoy rewrites it before forwarding. No backend code changes required.

#### Phase 4 — Cross-Namespace Routing with ReferenceGrant

**nginx can do:** Route only to Services in the **same namespace**. No cross-namespace support in the Ingress spec.

**Gateway API adds:** An HTTPRoute in `default` can route to a Service in `staging` — but only with explicit opt-in from the target namespace via `ReferenceGrant`. The target namespace owns its own security boundary.

```bash
kubectl apply -f gateway/06-cross-namespace.yaml
```

This creates in one file:
1. `staging` namespace
2. `gpen-api-v2` Deployment + Service in staging
3. `ReferenceGrant` in staging allowing HTTPRoutes from `default`
4. Updated HTTPRoute in default with cross-namespace backendRef

```yaml
# The explicit opt-in — staging namespace grants permission:
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-from-default
  namespace: staging          # lives in TARGET namespace
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: default        # only HTTPRoutes from default allowed
  to:
  - group: ""
    kind: Service             # to reference any Service in staging
```

```bash
# Hit /api 10 times — watch ~20% go to staging:
for i in $(seq 1 10); do curl -s -H "Host: gpen.local" http://localhost:9080/api; done
# 🔵 API v1 — stable (default)
# 🔵 API v1 — stable (default)
# 🟢 API v2 — canary (STAGING namespace — cross-namespace routing!)
# 🔵 API v1 — stable (default)
```

### 03-enhance-monitor.sh — Live Enhancement Monitor

**Terminal 2. Run during 03-enhance.sh.**

```bash
./scripts/03-enhance-monitor.sh
./scripts/03-enhance-monitor.sh --interval 0.5
```

Polls 4 checks every second — all against Envoy only (nginx is gone after cutover):

| Check | What it verifies |
|-------|-----------------|
| `GET /api` (no header) | Traffic distribution — v1 vs v2 vs staging |
| `GET /api` + `X-Version: v2` | Header routing — must ALWAYS return v2 |
| `GET /v1/products` | URL rewrite — body must match `/api` response |
| `HEAD /` | CORS — `Access-Control-Allow-Origin` header present |

Signal legend:

| Signal | Meaning |
|--------|---------|
| `v1 / v2 / staging` | Which backend served `/api` |
| `✔ → v2` | Header routing working |
| `✘ leaked v1` | Header routing broken |
| `✔ match` | URL rewrite working |
| `≠ diff` | URL rewrite not working |
| `no route` | HTTPRoute rule not applied yet |
| `✔ CORS` | Access-Control-Allow-Origin present |
| `no CORS yet` | CORS phase not applied yet (normal before Phase 1) |

## Kubernetes Manifests Deep Dive

### app/deployment.yaml

Three Deployments using `hashicorp/http-echo:latest`. Each responds with a fixed string on port 5678. The `-text` arg sets the response body.

```yaml
# gpen-web: 2 replicas — simulates a frontend service
# gpen-api-v1: 2 replicas — stable backend
# gpen-api-v2: 1 replica — canary backend (intentionally fewer replicas)
```

Why `http-echo`? Lightweight, no setup — every request returns a fixed string so you can instantly see which backend served each request by reading the response body.

### app/service.yaml

Three ClusterIP Services, each mapping port 80 to container port 5678. ClusterIP means these are not directly reachable from outside — all external traffic goes through Ingress or Gateway.

### ingress/ingress.yaml

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: gpen-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: gpen.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service: { name: gpen-web, port: { number: 80 } }
      - path: /api
        pathType: Prefix
        backend:
          service: { name: gpen-api-v1, port: { number: 80 } }
```

This is the "before" state. Note the `rewrite-target` annotation — a workaround required for nginx. Gateway API does the same thing natively in the spec.

### gateway/01-gateway.yaml

GatewayClass + Gateway. Platform team applies once. All dev teams share this Gateway — they only create HTTPRoutes.

### gateway/02-canary.yaml

HTTPRoute with `weight:` on both backendRefs. `weight: 90` / `weight: 10` = ~10% to v2. With nginx, this required two separate Ingress objects, canary annotations, and was unreliable across controller versions.

### gateway/03-cors.yaml

HTTPRoute with `ResponseHeaderModifier` filter adding 4 CORS headers. No annotation research needed — the fields are in the spec and work identically on every Gateway API implementation.

### gateway/04-header-routing.yaml

HTTPRoute with two rules for `/api`. The rule with `headers` match is evaluated first (Gateway API uses "first match wins" for rules within a route). Requests without the header fall through to the default rule.

### gateway/05-rewrite-timeout.yaml

HTTPRoute combining `URLRewrite` and `timeouts` on the same rule. `request: 3s` is the total client-to-response budget. `backendRequest: 2s` is how long Envoy waits for the upstream. These are per-rule, meaning different routes can have different timeouts in the same HTTPRoute.

### gateway/06-cross-namespace.yaml

One file creates five resources: a Namespace, a Deployment in staging, a Service in staging, a `ReferenceGrant` in staging, and an updated HTTPRoute in default. The ReferenceGrant is the security boundary — without it, the HTTPRoute's cross-namespace backendRef would be rejected by the controller with a `RefNotPermitted` status.

## Troubleshooting

### nginx returns 404 after setup

The Ingress resource may not be applied. Check:
```bash
kubectl get ingress -A
# If empty:
kubectl apply -f ingress/ingress.yaml
```

### Envoy Gateway stays `Programmed: False`

On kind, LoadBalancer services never get an external IP — this is normal. Use port-forward:
```bash
kubectl get svc -A -l "gateway.envoyproxy.io/owning-gateway-name=gpen-gateway"
kubectl port-forward -n <namespace> svc/<svc-name> 9080:80 &
```

### ingress2gateway produces no output

Check the Ingress exists and use the correct provider flag:
```bash
kubectl get ingress -A
ingress2gateway print --namespace=default --providers=ingress-nginx
```

### Port already in use (8080 or 9080)

```bash
pkill -f "kubectl port-forward"
```

### Wrong kubectl context

```bash
kubectl config use-context kind-gpen-demo
kubectl config current-context
# kind-gpen-demo
```

### HTTPRoute shows `RefNotPermitted`

Cross-namespace backendRef without ReferenceGrant. Apply:
```bash
kubectl apply -f gateway/06-cross-namespace.yaml
kubectl get referencegrant -n staging
```

### monitor shows GW_ISSUE (nginx=200, Envoy=404)

The HTTPRoute `parentRef` is not pointing to `gpen-gateway`. Check:
```bash
kubectl get httproute -o yaml | grep parentRefs -A5
# name should be: gpen-gateway
```

If it says `nginx` (from ingress2gateway's raw output), re-run the sed patch or apply `02-migrate.sh` again from Phase 2.

## Key Talking Points

| Topic | The Point |
|-------|-----------|
| **ingress-nginx retirement** | March 24, 2026 — no more patches. Every team relying on it is now running an unmaintained ingress controller |
| **Three-role model** | GatewayClass (cluster admin) → Gateway (platform team, once) → HTTPRoute (app dev, self-service). Separation of concerns baked into the API |
| **ingress2gateway** | Official CNCF tool. You don't rewrite migration YAML — the tool reads your live Ingress and outputs HTTPRoute. Proven at scale |
| **Zero downtime** | Both nginx and Envoy serve the same backends simultaneously during Phase 3. No maintenance window needed |
| **Monitor shows CUTOVER not MISMATCH** | After Ingress deletion, nginx=404 is expected and correct. The monitor understands this and shows green — not a false alarm |
| **weight: is native** | No annotation hacks. `weight: 90` on one backendRef and `weight: 10` on another — that's it |
| **ResponseHeaderModifier** | CORS in Gateway API is a first-class filter. Identical syntax on every implementation |
| **ReferenceGrant** | Cross-namespace routing with explicit consent from the target. The namespace that owns the Service decides who can route to it |
| **Portable** | Every feature shown works on Envoy Gateway, Cilium Gateway, Istio, HAProxy Ingress Controller — not vendor annotations |
| **Rollback is 5 seconds** | `kubectl apply -f ingress-backup.yaml` restores the old Ingress instantly. Gateway API resources are unaffected |
